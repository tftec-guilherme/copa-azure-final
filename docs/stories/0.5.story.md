### TD-6 — Rate limiting (15 min)

- [ ] `cd fifa2026-api && npm install express-rate-limit`
- [ ] Editar `fifa2026-api/src/index.js`:
  ```js
  const rateLimit = require('express-rate-limit');

  const generalLimiter = rateLimit({
    windowMs: 15 * 60 * 1000, // 15 min
    max: 100,
    message: { error: 'Muitas requisições. Aguarde alguns minutos.' },
    standardHeaders: true,
    legacyHeaders: false,
  });

  const loginLimiter = rateLimit({
    windowMs: 15 * 60 * 1000,
    max: 5,
    message: { error: 'Muitas tentativas. Aguarde alguns minutos.' },
    standardHeaders: true,
    legacyHeaders: false,
    skipSuccessfulRequests: true,  // só conta tentativas que falharam
  });

  // Aplicar antes das rotas
  app.use('/api/auth/login', loginLimiter);
  app.use(generalLimiter);
  ```

> **Cuidado com ordem dos middlewares:** rate limiters devem vir ANTES das rotas mas DEPOIS de `helmet`/`cors` para não bloquear preflight CORS.

> **Trust proxy:** atrás do iisnode/IIS, o IP visto pelo Express é `127.0.0.1` (proxy local). Adicionar `app.set('trust proxy', 1);` no início para o limiter usar o IP real do `X-Forwarded-For`.

## Dev Notes

**Por que `useEffect` em vez de `onError` no `useQuery`?**
- `onError` foi **deprecated no react-query v5** em favor do `meta.errorMessage` + global handler
- Para evitar mudar config global do QueryClient, `useEffect` observando `isError` é o caminho mais limpo e local

**Por que `app.disable('x-powered-by')` E web.config?**
- `app.disable('x-powered-by')` resolve o header gerado pelo Express (que `helmet` já remove)
- Em Azure App Service Windows + iisnode, o IIS adiciona o header de novo no nível do servidor
- A regra `<remove name="X-Powered-By" />` em `web.config` resolve no nível IIS

**Por que `skipSuccessfulRequests` no login?**
- Se o login funciona, não conta no limit (usuário válido pode logar várias vezes em sessões diferentes)
- Só conta falhas — protege contra brute force

**Por que `trust proxy`?**
- iisnode e Azure App Service ficam atrás de proxies (TCP forwarding)
- Sem `trust proxy`, todos os requests vêm de `127.0.0.1` e o rate limiter trataria como mesmo IP
- Com `trust proxy: 1`, o Express usa o `X-Forwarded-For` para identificar o IP real

### Causa raiz (TD-6 WAIVED)

express-rate-limit usa `MemoryStore` (in-process), e em Azure App Service o tráfego pode ser servido por múltiplos frontends/workers compartilhados. Cada request pode chegar em uma instância diferente do Node, com counter zerado. Plus: `req.ip` pode resolver para IPs internos rotativos. Diagnóstico confirmado por:
- Limiter está executando (header `RateLimit-Policy: 100;w=900` retornado)
- 105 requests sequenciais não acumularam (esperado: bloquear na 101ª)

### Soluções para iteração futura (novo tech debt — TD-7)

1. **Redis-backed rate limit:** `rate-limit-redis` + Azure Cache for Redis (~$16/mês). Counter compartilhado entre instâncias.
2. **Azure Front Door + WAF:** rate limiting na edge antes de chegar no backend.
3. **Aumentar SKU para Premium:** vem com VNet integration e mais previsibilidade de IP de cliente.

Para o evento, mitigação via CORS allowlist do frontend é suficiente. TD-7 fica para fase de hardening produção.

**Pontos fortes:**
- Order dos middlewares respeita preflight CORS antes do rate limiter
- `app.set('trust proxy', 1)` indispensável para iisnode + corretamente aplicado
- `skipSuccessfulRequests=true` no login limiter — mitigação anti brute force sem prejudicar UX legítima
- `standardHeaders: true, legacyHeaders: false` — RFC-compliant
- 5 useEffects com dependency arrays corretas (`[<error>]`); imports `useEffect` e `toast` adicionados; `isError` extraído de todos os 5 useQueries

**Observação:** login passa pelos DOIS limiters (loginLimiter + generalLimiter) — comportamento correto e desejado.
